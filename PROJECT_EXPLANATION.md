# Electricity Billing & Complaint Portal - Codebase Guide

This document explains the application from start to end so you can answer teacher questions about the project, architecture, code, database, and backlog coverage.

## 1. Project Purpose

The application is an Electricity Billing and Complaint Management Portal. It gives three roles:

- Customer: register, login, view bills, pay bills, download bills/receipts, raise complaints, and track complaint status.
- Admin: manage customers, link multiple consumer numbers, disconnect/reconnect customers, add bills, bulk upload bills, view complaints, and update complaint status.
- SME: view/search assigned complaints and update complaint status with remarks.

The goal is to keep billing, payment, customer management, and complaint handling in one simple web application.

## 2. Technology Stack

- Frontend: Angular 18 standalone component.
- Backend: Spring Boot 3.3.5 REST API.
- Database: H2 file database.
- Data access: Spring JDBC using `JdbcTemplate`.
- Validation: Jakarta Bean Validation annotations such as `@NotBlank`, `@Email`, and `@NotNull`.

Why this stack was used:

- Angular makes it easy to create role-based screens in one frontend.
- Spring Boot gives quick REST API development.
- H2 avoids separate database setup, useful for a PBL/demo project.
- JDBC keeps database logic easy to see in one file.

## 4. Backend Architecture

The backend has three main parts:

- `EbillingApplication`: starts Spring Boot.
- `ApiController`: exposes REST endpoints like login, add customer, pay bill, and update complaint.
- `Store`: contains database schema creation, seed data, queries, inserts, updates, and deletes.

Example:

```java
@RestController
@RequestMapping("/api")
@CrossOrigin(origins = "http://localhost:4200")
class ApiController {
  private final Store store;

  ApiController(Store store) {
    this.store = store;
  }
}
```

Why this is done:

- `@RestController` marks the class as an API controller.
- `@RequestMapping("/api")` keeps all API URLs under `/api`.
- `@CrossOrigin` allows Angular on port `4200` to call Spring Boot on port `8080`.
- `Store` is injected through the constructor.

## 5. Database Tables

The backend creates tables automatically when the app starts.

```java
jdbc.execute("""
    create table if not exists customers(
      id bigint generated by default as identity primary key,
      name varchar(120) not null,
      email varchar(160) not null unique,
      address varchar(255) not null,
      mobile varchar(20) not null,
      customer_type varchar(40) not null,
      electrical_section varchar(80) not null,
      connection_status varchar(30) not null
    )
    """);
```

Important tables:

- `customers`: customer profile details and connection status.
- `consumer_numbers`: one customer can have multiple consumer numbers.
- `users`: login users and roles.
- `bills`: monthly bill records.
- `payments`: payment receipts.
- `complaints`: complaint records and status.

Why `consumer_numbers` is a separate table:

- One customer can have many consumer numbers.
- This is a one-to-many relationship.
- It avoids storing comma-separated values inside the customer table.

## 6. Authentication Logic

Login checks email, password hash, and role.

```java
var rows = jdbc.query("""
    select id, name, email, role, customer_id from users
    where lower(email) = lower(?) and password_hash = ? and lower(role) = lower(?)
    """, mapper, request.email(), hash(request.password()), request.role());
```

Why passwords are hashed:

- Plain passwords should not be stored.
- The app uses SHA-256 hashing before saving/checking passwords.

```java
private String hash(String value) {
  return HexFormat.of().formatHex(
      MessageDigest.getInstance("SHA-256").digest(value.getBytes())
  );
}
```

For a production system, a stronger password hashing algorithm like BCrypt would be better.

## 7. Customer Management

Admin can now see complete customer details:

- Customer ID.
- Name.
- Email.
- Mobile.
- Address.
- Customer type.
- Electrical section.
- Connection status.
- All linked consumer numbers.

The customer response object includes all these fields:

```java
class Customer {
  public long id;
  public String name;
  public String email;
  public List<String> consumerNumbers;
  public String connectionStatus;
  public String address;
  public String mobile;
  public String customerType;
  public String electricalSection;
}
```

Admin can also delete a customer. Because the database foreign keys use `on delete cascade`, linked consumer numbers, bills, payments, and complaints are removed with that customer.

```java
void deleteCustomer(long customerId) {
  requireCustomer(customerId);
  jdbc.update("delete from customers where id = ?", customerId);
}
```

Why confirmation is used in the UI:

- Delete is a destructive action.
- The admin should confirm before customer data is removed.

## 8. Bill Flow

Customer bill flow:

1. Customer logs in.
2. Angular loads bill summary, bills, payments, and complaints.
3. Customer filters bills by status.
4. Customer pays an unpaid bill.
5. Backend marks bill as paid and creates a payment record.
6. Customer can download bill/payment details.

Important backend code:

```java
jdbc.update("update bills set status = 'PAID' where id = ?", billId);
jdbc.update("""
    insert into payments(customer_id, bill_id, amount, card_last4, paid_at)
    values (?, ?, ?, ?, ?)
    """, customerId, billId, bill.amount, request.cardLast4(), Timestamp.valueOf(paidAt));
```

Why this is done:

- Bill status changes from `UNPAID` to `PAID`.
- A payment receipt is saved separately so payment history is available.

## 9. Complaint Flow

Customer complaint flow:

1. Customer submits complaint title, department, and description.
2. Backend saves it with status `OPEN`.
3. Admin or SME can filter complaints.
4. Admin/SME updates status and remarks.
5. Customer can view the updated complaint status.

Example update:

```java
jdbc.update("update complaints set status = ?, remarks = ? where id = ?",
    request.status(), request.remarks(), complaintId);
```

Why remarks are stored:

- Remarks explain what action was taken.
- This helps customer and staff understand complaint progress.

## 10. Frontend Architecture

The Angular frontend uses:

- `app.component.ts`: state, forms, API calls, and actions.
- `app.component.html`: screens and role-based UI.
- `app.component.css`: component styling.
- `styles.css`: global styling.

Role tabs are computed from the logged-in user:

```typescript
roleTabs = computed(() => {
  const role = this.user()?.role;
  if (role === 'ADMIN') {
    return ['dashboard', 'customers', 'bills', 'complaints'];
  }
  if (role === 'SME') {
    return ['dashboard', 'complaints'];
  }
  return ['dashboard', 'bills', 'payments', 'complaints'];
});
```

Why this is useful:

- Customer, Admin, and SME see only their own screens.
- The app stays simple without using a routing module.

## 13. Common Teacher Questions

Question: Why use H2 database?

Answer: H2 keeps setup simple for a PBL project. The database runs locally and persists to `backend/data/ebilling-db`, so no MySQL/PostgreSQL installation is required.

Question: How is one customer linked to many consumers?

Answer: The `consumer_numbers` table has `customer_id` as a foreign key. Multiple rows can point to the same customer ID.

Question: What happens when a customer is deleted?

Answer: The customer row is deleted. Because related tables use `on delete cascade`, linked consumer numbers, bills, payments, complaints, and customer login are also removed.

Question: How are roles separated?

Answer: The login response includes the role. Angular uses that role to show only the correct tabs and actions.

Question: How are validation errors handled?

Answer: Request records use validation annotations. The backend also throws `IllegalArgumentException` for duplicate email or consumer number, and the Angular toast displays the error message.

Question: Why is the backend in one Java file?

Answer: For a small sprint/demo project, keeping controller, repository, DTOs, and models in one file makes the flow easy to read. In a larger production project, these would be split into packages.

Question: What is the most important architecture idea?

Answer: Angular handles UI state and API calls, Spring Boot exposes REST endpoints, and H2 stores persistent data. Each role uses the same API layer but sees different UI actions.
